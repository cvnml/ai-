# Implementation Review

这是一个通用代码实现审查技能。它不负责重新设计方案，也不默认重写代码；它的目标是在提交“完成”之前，像 reviewer 一样找出实现链路里的遗漏。

适用场景：

- 用户要求 review、复查、检查实现是否完整。
- 已经写完功能，但需要确认有没有隐藏边界。
- 涉及导入、批量回填、默认选中、表单提交、树/表格/弹窗、父子组件联动、异步加载、确认链路复用。
- 用户问“这样真的会生效吗”“是不是还差一步”“有没有风险点”。

不适用场景：

- 纯代码格式调整。
- 单行文案修改。
- 用户明确只要解释，不要检查代码。
- 还没有具体实现时的产品方案讨论；这种情况先用计划或澄清。

## 审查原则

不要只检查“有没有调用函数”。要按真实用户路径审查：

```text
用户操作 → 输入校验 → 数据转换 → 接口请求 → 返回处理 → 组件状态 → UI 可见结果 → 父组件同步 → 最终确认/提交 → 清理恢复
```

如果其中某一步依赖异步加载、懒加载、弹窗打开时机、子组件 ref、接口返回结构，就必须明确检查。

## 通用审查清单

### 1. 入口和触发是否正确

检查：

- 用户点击的按钮、菜单、弹窗 footer 是否绑定到正确方法。
- 当前 tab、当前模式、当前类型是否会影响方法分支。
- 禁用、loading、防重复点击是否会阻断正常路径。
- 入口是否只在应该出现的场景出现。

输出时说明：

- 入口在哪个文件、哪一行。
- 触发后进入哪个方法。
- 有哪些分支条件。

### 2. 输入校验是否覆盖真实边界

检查：

- 必填项是否校验。
- 文件、表单、选择项、当前 tab 是否校验。
- 空数组、空字符串、重复数据、无效字段是否处理。
- 错误提示是否会提前 return，避免继续执行错误请求。

不要只说“有校验”，要说缺哪个边界。

### 3. 数据转换是否和下游字段一致

检查：

- 前端读取字段和接口需要字段是否一致。
- 接口返回字段和组件内部字段是否一致。
- 是否需要把后端字段转换成 UI 组件需要的 `id`、`label`、`children`、`leaf`、`disabled` 等。
- 是否存在“Excel/导入文本”和“后端真实对象”的区别。

审查时要特别问：

```text
现在传下去的是用户输入的原始值，还是后端匹配后的真实对象？
```

### 4. 接口调用是否只是中间步骤

检查：

- 请求参数是否包含当前业务分支需要的类型字段。
- 返回成功后是否真的进入后续 UI/状态处理。
- 接口失败、空返回、部分返回是否有提示。
- 是否错误地把“接口成功”等同于“功能完成”。

关键规则：

```text
接口成功不是终点，用户可见状态和后续提交链路成功才是终点。
```

### 5. 组件状态和 UI 可见状态是否同时成立

这是最重要的审查点。

很多 bug 是“状态已经写了，但界面没显示”。审查时必须分开问：

- 数据是否进入组件内部状态。
- 数据是否进入用户能看到的 DOM/树/表格/列表。
- 数据是否被 Element UI、虚拟列表、懒加载树、分页表格实际渲染。
- 默认选中是否只是数组里有数据，还是 UI 勾选也同步了。

典型风险：

- 树节点未加载，无法 append 到父节点。
- 表格分页不在当前页，看不到新增项。
- 弹窗关闭后状态丢失。
- 子组件内部状态更新了，但没有 emit 给父组件。

### 6. 懒加载和异步生命周期是否有补偿机制

如果组件依赖 lazy load、接口加载、ref、弹窗打开后渲染，必须检查：

- 目标父节点当前不存在时怎么办。
- 是否需要缓存待处理数据。
- 节点加载完成后是否会重试挂载。
- `$nextTick` 是否只是解决渲染时机，而不是解决数据不存在问题。
- 清空/关闭/重新打开时是否清理缓存。

审查规则：

```text
如果当前做法是“找得到节点就处理，找不到就跳过”，必须追问找不到时数据是否会丢、界面是否会不一致。
```

### 7. 父子组件同步是否闭环

Vue2 里尤其要检查：

- 子组件是否 `$emit` 给父组件。
- 父组件接收事件后是否更新自己的提交数组/表单字段。
- `.sync`、`v-model`、自定义事件名是否对应。
- 通过 `$refs` 调子组件方法后，子组件内部状态是否也同步给父组件。

不要只看子组件状态，要看父组件最终提交读取的是哪个变量。

### 8. 原有确认/提交链路是否真的复用

如果需求说“导入后仍走原确定按钮/原保存流程”，审查：

- 导入结果是否进入原确定按钮读取的数据源。
- 原确定按钮是否会清空、覆盖、重新计算这些数据。
- 原确认接口是否需要再次补充字段。
- 事件总线、父组件回调、最终表单列表是否还能收到导入结果。

关键问题：

```text
用户完成新入口操作后，再按原来的确定/保存按钮，代码读取的是新数据还是旧数据？
```

### 9. 清理和重复操作是否安全

检查：

- 关闭弹窗是否清空临时文件、loading、缓存。
- 重新打开是否会残留上一次导入结果。
- 多次导入是否去重。
- 切 tab 后是否把数据导入到错误的组件。
- 取消操作是否不污染主表单。

### 10. 验证方式是否覆盖真实用户路径

审查完成后给出验证建议：

- 最小静态验证：lint/test。
- 组件级验证：mock 接口返回，检查状态、emit、DOM。
- 浏览器验证：按真实点击路径走一遍。
- 对关键边界至少验证一个失败路径。

不要把“测试通过”写成笼统结论；要说明测了哪条路径，没测哪条路径。

## 输出格式

审查结果用下面结构：

```markdown
## Review 结论

- 结论：通过 / 有风险 / 不完整
- 最大风险点：...

## 链路核对

1. 用户入口：...
2. 输入校验：...
3. 数据转换：...
4. 接口调用：...
5. 状态更新：...
6. UI 可见性：...
7. 父子组件同步：...
8. 原确认/提交链路：...
9. 清理逻辑：...

## 必须修的问题

- ...

## 建议补强的问题

- ...

## 验证建议

- ...
```

如果没有发现问题，也要明确列出“已核对的链路”，不要只说“看起来没问题”。

## 特别提醒

当功能涉及树、表格、弹窗、导入、批量回填、默认选中时，必须同时检查：

```text
业务状态是否正确 + 用户界面是否可见 + 最终提交是否读到同一份数据
```

这三者任何一个没闭环，都不能判断功能真正完成。
