# Execution Checkpoint

## 目标

`execution-checkpoint` 负责在执行过程中生成轻量 checkpoint，让用户知道当前做到哪里、是否偏离计划、下一步是什么。

它解决的问题是：执行长任务时，如果只在最后总结，用户很难知道中途是否偏离计划；但如果把每次小改动都写成长期文档，又会造成文档抖动和 token 浪费。这个 Skill 只做阶段性状态摘要，不做过程流水账。

## 一句话边界

只做“执行状态 checkpoint”，不改代码，不生成新计划，不做长期文档对账，不替代实现审查。

## 什么时候使用

当出现以下情况时使用：

- 一个阶段完成。
- 执行中断前。
- 出现计划外约束。
- 验证方式降级。
- 依赖、构建方式、模块边界发生变化。
- 用户问“现在进度到哪了”。
- 执行方准备暂停，想留下恢复入口。

## 什么时候不要使用

不要在这些场景使用：

- 用户要生成或细化计划，应使用 `plan-mode-plus`。
- 用户要把已批准计划落成本地文档，应使用 `modular-plan-docs`。
- 用户准备修改生产代码前要确认施工图，应使用 `edit-node-blueprint`。
- 用户要审查已实现代码链路，应使用 `implementation-review`。
- 用户要设计或执行前端测试，应使用 `frontend-test`。
- 用户要求实现后文档对账；当前应单独规划 doc reconcile 类 Skill，不由本 Skill 承接。

## 工作流程

固定流程：

```text
确认当前计划 / 阶段
→ 汇总已完成事项
→ 汇总验证结果
→ 检查是否偏离已批准计划
→ 说明阻塞和风险
→ 给出下一步建议
→ 停止
```

本 Skill 完成后不自动继续编码。

## checkpoint 内容

输出时应包含：

1. 当前执行的计划或阶段。
2. 已完成事项。
3. 当前阻塞或风险。
4. 已运行的验证和结果。
5. 与已批准计划相比是否发生偏离。
6. 下一步建议。
7. 是否需要暂停、继续、回到 `plan-mode-plus` 细化，或后续触发文档同步。

## 输出格式

默认输出：

```markdown
## Execution Checkpoint

### 当前阶段

- <当前计划 / 阶段 / 本轮目标>

### 已完成

- <完成项 1>
- <完成项 2>

### 验证结果

- <已运行命令、测试、页面验证或未运行原因>

### 偏离计划情况

- <无偏离 / 有偏离及原因>

### 阻塞与风险

- <阻塞或风险；没有则写“无明显阻塞”>

### 下一步

- <建议下一步动作>
```

输出要短，重点是恢复和决策，不要写成日志流水账。

## 偏离计划判断

出现以下情况时，应明确写“偏离计划”：

- 实际修改文件超出已批准范围。
- 阶段目标发生变化。
- 验证方式从自动化降级为仅静态检查或未验证。
- 发现新依赖、构建策略或模块边界变化。
- 用户确认过的不做项被重新触碰。

偏离计划时，不要继续默默执行，应建议回到 `plan-mode-plus` 或请求用户确认下一步。

## 不写入长期文档的内容

checkpoint 默认不要写入项目长期文档，也不要创建文件记录，除非用户明确要求。

不要记录：

- 每次测试失败的流水账。
- 临时环境问题细节。
- 命名反复调整过程。
- 中间尝试过但已放弃的小改法。
- 大段命令输出。

这些内容适合留在对话里，不适合默认进入长期文档。

## 与其他 Skills 的关系

### plan-implementation-runner

runner 在阶段完成、阻塞、偏离计划或验证降级时，可以触发本 Skill 输出 checkpoint。

### plan-mode-plus

如果 checkpoint 发现计划不够细或阶段边界变化，应建议回到 `plan-mode-plus` 继续细化。

### modular-plan-docs

本 Skill 不做项目本地计划快照、索引或归档。只有当用户明确要求把已批准计划落地或归档时，才使用 `modular-plan-docs`。

### implementation-review

本 Skill 不做完整实现审查。如果用户问“实现有没有漏链路”，使用 `implementation-review`。

## 反模式

禁止：

- 把 checkpoint 写成完整过程日志。
- 用 checkpoint 替代实现审查。
- 用 checkpoint 替代计划细化。
- 在 checkpoint 中修改代码或文档。
- 创建 `task_plan.md`、`findings.md`、`progress.md`。
- 明明发生计划偏离，却只写“进展正常”。

## 验证方式

### 结构验证

检查 `SKILL.md` 是否满足：

- frontmatter 有 `name: execution-checkpoint`。
- description 明确阶段状态、阻塞、偏离计划、验证结果和下一步。
- 正文明确不改代码、不生成新 plan、不做长期文档对账。
- 正文明确 checkpoint 不是过程日志。
- 正文明确偏离计划时要停下建议回到计划或确认。

### 行为 dry run

1. “现在做到哪了，下一步是什么？”
   - 应输出当前阶段、已完成、验证、偏离计划、风险和下一步。
   - 不应重写计划。

2. “测试没跑完但要继续。”
   - 应明确验证降级或验证不足。
   - 不应假装完成。

3. “改动超出计划范围了。”
   - 应标记偏离计划。
   - 应建议回到 `plan-mode-plus` 或请求确认。
