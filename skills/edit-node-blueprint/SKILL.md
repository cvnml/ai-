# Edit Node Blueprint

## 目标

`edit-node-blueprint` 负责在每一次独立写文件动作前，把本次写入节点的施工图摊开给用户看。

它解决的问题是：旧的执行器容易把多个独立 Edit/Write 合并成一次超大提问，用户无法准确判断每一次写入会造成什么影响。这个 Skill 要求“一次问题工具对应一个写文件节点”，让用户逐次校正即将写入的文件、函数、数据流、副作用和风险边界。

## 一句话边界

只做“写文件节点级施工图确认”，不编码，不生成 plan，不创建项目本地计划，不做 checkpoint，不做文档对账。

## 什么时候使用

当出现以下情况时使用：

- 准备调用 `Edit` / `Write` / `NotebookEdit` 修改生产代码前。
- 用户说“这次 Edit 前先把要改的函数讲清楚”。
- 用户说“不要把多个 Edit 合并成一个问题”。
- 执行过程中出现计划外新增写入节点，需要写前确认。
- 某次写入会影响权限、持久化、外部 API、删除数据、进程关闭、构建配置或用户可见交互。

## 什么时候不要使用

不要在这些场景使用：

- 用户只是要生成或细化计划，应使用 `plan-mode-plus`。
- 用户要把已批准计划落成本地文档，应使用 `modular-plan-docs`。
- 用户要查看执行进度，应使用 `execution-checkpoint`。
- 用户要审查已实现代码链路，应使用 `implementation-review`。
- 用户要设计或执行前端测试，应使用 `frontend-test`。
- 普通测试断言或 fixture 小改动，且不引入新测试框架、不改变业务/API 行为、不改变测试策略。

## 工作流程

固定流程：

```text
确认本次写文件节点
→ 识别本次要改的文件和关键函数
→ 判断每个函数在业务链路里的真实分工
→ 设计动态问题/tabs
→ 使用 AskUserQuestion 展示施工图判断
→ 用户确认无偏差后停止
```

本 Skill 完成后只表示“本次写文件节点施工图已确认”。它本身不调用写文件工具。

## 写文件节点定义

一个写文件节点通常是：

- 一次独立 `Edit`。
- 一次独立 `Write`。
- 一组必须在同一次写入里完成、职责紧密相关的函数修改。

不是一个写文件节点的情况：

- 三个不同文件的独立改动。
- 三个不同职责的独立 Edit。
- 一个完整阶段里的所有代码改动。
- 一个长期计划里的多个阶段。

如果是三个独立 Edit，就必须调用三次问题工具。

## AskUserQuestion 使用规则

一次 `AskUserQuestion` 只对应一个写文件节点。

同一次 `AskUserQuestion` 里的多个问题/tabs，只用于拆解当前节点内部的偏差点。

当本节点属于一个已批准的大功能或连续执行范围时，问题和推荐选项必须明确说明：“当前只确认这个写文件节点，不代表功能目标缩水；该节点完成后由 runner 继续下一个已批准节点”。不要把“只做当前节点”包装成“第一版最小完成”或“先打骨架”的推荐范围。

必须把关键施工图写在用户一眼可见的问题正文和选项描述里。不要只放在 preview 里。

问题应围绕“哪里可能有偏差”设计，例如：

- 函数职责是否偏了。
- 数据来源或输出对象是否偏了。
- 副作用边界是否偏了。
- 错误兜底是否偏了。
- 安全边界是否偏了。
- 验证方式是否偏了。

不要问泛化审批问题：

- “是否继续？”
- “是否同意？”
- “目标是否正确？”
- “还有问题吗？”

## 动态 tabs 设计规则

tabs/问题不能机械套同一组模板。要根据函数在当前链路里的分工动态生成。

### 入口协调函数

重点问：

- 触发入口是否完整。
- 是否会重复触发。
- 是否需要防重入。
- 失败后是否继续。
- 是否会影响用户可见流程。

### 进程或资源清理函数

重点问：

- 清理范围是否只覆盖 owned 资源。
- 是否存在误杀、误删、误关风险。
- 超时后怎么兜底。
- 已退出或不存在时怎么处理。
- 日志和错误是否足够排查。

### 数据转换函数

重点问：

- 字段从哪里来。
- 转换后给谁用。
- 空值、重复值、异常值怎么处理。
- 是否会丢失业务字段。
- 是否和接口/组件需要的结构一致。

### 接口请求函数

重点问：

- 请求参数是否包含当前业务分支需要的字段。
- 成功返回后是否进入用户可见状态更新。
- 失败、空返回、部分返回怎么提示。
- 是否错误地把“接口成功”当成功能完成。

### UI 状态函数

重点问：

- 状态是否真的渲染到界面。
- 父子组件是否同步。
- 异步加载、懒加载、弹窗打开时机是否会丢数据。
- 关闭或重复操作是否会残留旧状态。

### 测试函数

重点问：

- 测试层级是单元测试、组件测试还是集成测试。
- mock/stub 边界是否正确。
- 断言目标是否覆盖真正风险。
- 是否引入真实进程、网络、文件等副作用。
- 是否需要新测试工具；如果需要，必须说明影响。

## 输出要求

默认使用 `AskUserQuestion`，不要只用普通文本总结。

提问结构建议：

1. 先说明本次只准备修改哪个文件或写入节点。
2. 再说明本次节点的职责。
3. 按函数分工动态拆成 1-4 个问题。
4. 每个选项都说明选择后代表什么偏差或无偏差。
5. 如果是大功能的连续实现，推荐项要写成“当前节点范围正确，后续节点继续”，而不是“只做最小版本”。

如果用户指出偏差，停止并让执行方调整施工图；不要继续推进写文件。

## 与其他 Skills 的关系

### plan-mode-plus

`plan-mode-plus` 负责把当前阶段计划细化到可编码。本 Skill 只负责某次写文件节点前的施工图确认。

### plan-implementation-runner

`plan-implementation-runner` 可以在每个独立写文件节点前调用本 Skill。用户确认后，runner 才继续写入。

### execution-checkpoint

`execution-checkpoint` 负责阶段完成、阻塞或偏离时的状态记录。本 Skill 不记录进度。

### implementation-review

`implementation-review` 负责实现完成后的链路审查。本 Skill 是写入前确认，不替代 review。

## 反模式

禁止：

- 把多个独立 Edit 合并成一次问题工具。
- 把整个阶段的所有改动一次问完。
- 每个函数都套同一组 tabs。
- 只把关键施工图放在 preview。
- 只问“是否继续 / 是否同意”。
- 用户指出偏差后仍继续写文件。
- 在本 Skill 内编码、生成计划或做 checkpoint。

## 验证方式

### 结构验证

检查 `SKILL.md` 是否满足：

- frontmatter 有 `name: edit-node-blueprint`。
- description 明确“一次问题工具对应一个写文件节点”。
- 正文明确多个独立 Edit 要多次提问。
- 正文明确 tabs 按函数分工动态生成，不固定模板。
- 正文明确不编码、不生成 plan、不创建本地计划、不做 checkpoint。

### 行为 dry run

1. “准备改 A 文件前，把这次 Edit 要改的函数讲清楚。”
   - 应只展示 A 文件这次写入节点。
   - 不应顺便塞 B/C 文件。

2. “这次要改三个文件，各自都是独立 Edit。”
   - 应要求三次写文件节点级施工图确认。
   - 不应一次问题工具里塞三个独立 Edit。

3. “一个函数是退出入口，一个函数是进程清理，一个函数是测试断言。”
   - 三者问题重点应不同。
   - 不应都套同一组固定 tabs。
