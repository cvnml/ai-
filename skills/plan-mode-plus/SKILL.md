# Plan Mode Plus

## 目标

`plan-mode-plus` 是 Claude Code 原生 plan mode 的增强版。

它解决的问题是：原生 plan mode 可以生成计划文档，但如果过早写计划，容易在没有真正理解需求、没有读现有代码、没有确认关键分叉的情况下，把猜测写进计划。这个 Skill 要求在写入原生 plan mode 当前提供的计划文件前，先澄清目标、只读调研、查询必要官方文档，并和用户确认关键选择。

它还解决另一个问题：计划如果只停留在“阶段路线图”，后续执行阶段就会被迫继续补设计、频繁拆节点和反复确认。`plan-mode-plus` 应支持读取上一版 plan，按阶段 review 和细化，把当前要进入编码的阶段细化到足够连续实现的程度。

最终产物是一份轻量但可审批的原生 plan 文档，用来等待用户批准或修改。计划文档的目标必须覆盖用户提出的完整功能和完整需求；阶段拆分只是实现组织方式，不是把需求目标缩水成最小片段。

## 一句话边界

只做“调研充分、覆盖完整需求、可审批、可按阶段细化的 plan 文档”，不做“项目计划文档管理”，不编码，不创建项目本地 `plans`，不改 `CLAUDE.md`，不执行计划。

## 什么时候使用

当用户表达以下意图时使用：

- “先做计划”。
- “进入 plan mode”。
- “不要写代码，先规划”。
- “先调研再给方案”。
- “生成实施计划”。
- “设计实现计划”。
- “继续细化这个计划”。
- “review 一下上次 plan 能不能直接开写”。
- “把某个阶段细化到可以一次性编码”。
- “这个 plan 还太粗，继续完善”。
- 任务复杂，需要理解已有代码后才能计划。
- 涉及多模块、多技术路线、第三方库、SDK、API、CLI。
- 用户希望得到一份可审批的计划文档。

## 什么时候不要使用

不要在这些场景使用本 Skill：

- 用户只是问概念，不需要计划文档。
- 用户明确要求直接实现，且任务很小、边界清晰。
- 用户要求创建项目本地 `plans/`、模块计划、计划索引或归档。
- 用户要求执行已有计划。
- 用户要求持续 checkpoint、文档回写或实现审计。
- 用户只想澄清问题现象，不想进入计划生成。
- 用户要求从日志沉淀 Skill。

如果用户要求“管理项目本地计划文档、模块计划、计划索引、归档”，这是后续独立的模块化 plan 管理 Skill 范围，不属于 `plan-mode-plus`。

## 工作总流程

固定流程：

```text
澄清目标 → 只读调研 → 查询必要官方文档 → 提问确认关键分叉 → 写入或更新原生 plan mode 当前计划文件 → 等待用户批准/修改
```

如果用户是在继续上一版 plan 或要求达到可编码精细度，使用阶段级细化流程：

```text
读取上一版 plan → 保留总阶段地图 → 确认当前细化阶段 → review 当前阶段成熟度 → 用问题工具展示判断 → 继续调研/提问/细化当前阶段 → 写入原生 plan mode 当前计划文件 → 等待用户批准/修改
```

完成 plan 文档后必须停下，不要自动进入编码。

## 第 1 步：澄清真实目标和边界

开始时先用业务语言复述用户想达成的结果。

如果用户目标、范围、验收结果不清楚，要先提问。提问要通俗，不能把变量名、函数名、英文缩写或内部术语直接当成解释。

如果用户提出的是大功能，不要默认要求用户先缩小成“最小可落地边界”。应先保留完整功能目标和完整需求，再把它拆成阶段路线；Phase 1 是第一个可闭环阶段，不是让用户放弃后续功能。只有真实目标、验收结果或外部集成选择不清楚时，才把边界问题作为阻塞提问。

需要确认的内容通常包括：

- 用户真正想解决的问题或达成的结果是什么。
- 本轮是否只生成或细化计划，不写代码。
- 本轮包含什么，不包含什么。
- 用户期望怎样才算完成。
- 这是新计划，还是继续完善当前原生 plan mode 已经指定的计划草稿。
- 如果是继续计划，本轮要细化的是哪一个阶段。

提问一次最多 1-4 个问题，避免像审问表一样把用户压垮。

## 第 2 步：只读调研项目现状

写 plan 之前必须按任务需要做只读调研。

优先调研：

- 相关源码。
- 已有 `CLAUDE.md`、README、设计文档或旧计划草稿。
- 同类功能、工具函数、组件模式、API 调用模式。
- 现有测试文件、测试命令、手动验证入口。
- 模块边界、数据流、状态流、接口约束。

这一阶段禁止：

- 修改项目源码。
- 创建项目本地计划文件。
- 修改 `CLAUDE.md`。
- 安装、升级或删除依赖。
- 运行会改变环境、生成文件或写入数据的命令。

如果找不到足够上下文，计划里必须写明“上下文不足”，不要假装已经调研充分。

## 第 3 步：涉及第三方时查官方文档

如果计划涉及库、框架、SDK、API、CLI 或云服务，必须查询当前文档。

优先顺序：

1. 使用 Context7 查询库、框架、SDK、CLI 的当前文档。
2. 如果 Context7 不覆盖，再使用官方文档页面。
3. 只把和当前任务有关的关键结论写入 plan。

不要把过期经验直接写进计划。

如果不涉及第三方文档查询，计划里写明：

```text
不涉及第三方文档查询。
```

## 第 4 步：识别关键方案分叉并提问

写最终 plan 前，必须判断是否存在关键方案分叉。

需要提问的情况：

- 有两个以上合理技术路线。
- 会影响安全、权限、数据保存、接口语义、依赖引入、构建方式。
- 会改变用户可见交互。
- 会改变现有模块边界或数据流。
- 范围或验收标准不清。

提问方式：

- 每个选项都先用普通语言解释“这是什么”。
- 再说明它放到当前项目里会发生什么。
- 再说明好处、代价和推荐项。
- 用户确认后再写进 plan。

如果没有真实分叉，计划里要写明：

```text
未发现需要用户提前选择的关键方案分叉，原因是当前项目已有明确同类模式。
```

## 第 5 步：阶段级计划成熟度 review

当用户要求继续上一版 plan、review 某个阶段能不能进入编码、或把某个阶段细化到可一次性编码时，必须执行本步骤。

### 核心原则

- 保留总阶段地图。
- 大功能默认按阶段拆解，不把“最小可落地边界”当成额外前置选择。
- 每轮优先细化当前要进入编码的阶段；如果用户要求整份计划达到连续执行精细度，可以逐阶段标记多个“可编码”阶段。
- 未轮到且不在本轮批准执行范围内的阶段保持粗粒度，不提前写函数级施工图。
- “一次性编码”指已批准且已标记可编码的执行范围内连续完成，可以是单阶段，也可以是多个成熟阶段。
- review 过程要多用问题工具展示模型判断，让用户校正偏差。

### 当前阶段成熟度检查项

review 当前阶段时，检查：

1. 阶段目标是否明确。
2. 阶段编码范围是否明确。
3. 阶段不做项是否明确。
4. 预计修改文件是否明确。
5. 关键函数 / 方法 / 数据流是否明确到足够指导编码。
6. 入参、出参、副作用、错误流是否覆盖高风险节点。
7. 用户可见交互变化是否明确。
8. 需要调用的第三方库、SDK、框架 API 是否已查当前文档。
9. 验证方式是否足够支撑阶段完成判断。
10. 是否仍有未确认关键分叉。

### 每轮 review 必须用问题工具展示判断

出现以下情况时，必须使用 `AskUserQuestion`，不能只写普通文本：

- 选择当前要细化哪个阶段。
- 判断上一版 plan 是否足够进入编码。
- 发现阶段边界可能不对。
- 发现文件、函数、数据流、验证方式不够明确。
- 发现多个合理细化方向。
- 准备把某个阶段标记为“可编码”。

一次复杂 review 不要只问“这个计划能不能进入编码”。应按可校正的判断点拆成多个问题，例如：

1. 当前阶段选择是否正确。
2. 阶段目标和不做项是否正确。
3. 文件和关键函数范围是否足够明确。
4. 数据流、副作用、错误流是否还有遗漏。
5. 验证方式是否足够判断阶段完成。
6. 是否允许把该阶段标记为可编码。

每个问题都要先用普通语言解释“这是什么”，再说明“放到当前项目里意味着什么”，最后给出用户可选择的偏差类型。

问题工具展示的是 Claude 已经做出的 review 判断，让用户校正偏差。不要把问题写成让用户替 Claude 设计方案。

好例子：

- “我判断 Phase 1 现在还不能编码，因为只写了页面目标，没有写清数据从哪里来。这个判断有没有偏差？”
- “我准备把 Phase 2 标记为可编码，因为文件、关键函数、错误兜底和验证入口都已明确。你主要担心哪类偏差？”

坏例子：

- “你想怎么设计？”
- “要不要继续？”
- “是否同意？”
- “这个阶段还有问题吗？”

### 成熟度结论

review 后给出三类结论。

#### A. 当前阶段可以进入一次性编码

含义：当前阶段计划已经足够具体，执行阶段不需要继续补大设计。

要求：

- 文件范围清楚。
- 关键函数和数据流清楚。
- 高风险副作用和错误流清楚。
- 验证方式清楚。
- 没有未确认关键分叉。
- 已通过问题工具让用户看过关键判断，并且用户没有指出偏差。

后续：

- 更新 plan，把该阶段标记为“可编码”。
- 如果多个阶段都已细化成熟，可在 plan 中标明“可连续执行范围”。
- 等待用户批准当前执行范围。
- 批准后可交给执行流程连续实现。

#### B. 当前阶段需要继续细化 plan

含义：总方向正确，但当前阶段还不能支撑一次性编码。

常见原因：

- 只写了阶段目标，没有写清具体文件和关键函数。
- 没有说明数据从哪里来、给谁用。
- 没有说明副作用和错误兜底。
- 没有明确测试/验证入口。
- 有多个方案分叉未确认。

后续：

- 用问题工具展示缺口判断。
- 继续只读调研。
- 必要时提问。
- 更新原生 plan mode 当前计划文件中的当前阶段。
- 不进入编码。

#### C. 需要重新澄清总目标或阶段边界

含义：当前阶段的目标、范围或验收结果本身还不稳定，或者阶段切分不合理。

后续：

- 用问题工具说明哪里不稳定。
- 回到澄清目标或阶段边界。
- 必要时调整总阶段地图。
- 不做实现细化。
- 不进入编码。

## 第 6 步：生成轻量 plan 文档

只允许创建或更新原生 plan mode 当前指定的计划文件。

路径规则：

1. 如果当前处于 Claude Code plan mode，并且系统提示里提供了 `Plan File Info`，必须使用该提示给出的完整计划文件路径。
2. 不要硬编码 `C:\Users\admin\.claude\plans\*.md`、`~/.claude/plans/*.md` 或任何自猜路径，因为不同 Claude Code 版本或不同项目上下文可能会把 plan 文档放到项目隔离目录下。
3. 如果当前没有系统提供的 plan 文件路径，不要自己猜路径；应先进入原生 plan mode，或向用户确认应该使用哪一个原生 plan 文件路径。

禁止创建：

- 项目根 `plans/*.md`
- 模块目录 `plans/*.md`
- `PLAN.<feature>.md`
- `task_plan.md`
- `findings.md`
- `progress.md`
- atomic task cards

计划文档要轻量，写阶段路线和关键事实；当前细化阶段可以写到文件、关键函数、数据流、副作用、错误流和验证方式，但不要写完整代码。

## 计划文档模板

生成原生 plan mode 当前指定的计划文件时使用这个结构：

```markdown
# Plan: <任务名>

## 1. 用户真实目标

<用业务语言说明用户想完整达成的功能和需求结果；不要只写 Phase 1 或最小闭环。>

## 2. 本轮范围

### 包含

- <本轮会规划或细化的内容>

### 不包含

- 不编码
- 不创建项目本地 plans
- 不修改 CLAUDE.md
- 不做 doc reconcile
- 不执行 atomic implementation
- <其他本轮明确不做的内容>

## 3. 已调研事实

### 项目现状

- <读到的相关目录、模块、现有模式>

### 现有实现 / 类似参考

- <已有代码里可复用或需要保持一致的模式>

### 测试与验证方式

- <现有测试命令、测试文件、手动验证入口；如果暂未找到，要写明>

### 第三方官方文档结论

- <涉及第三方库/SDK/API/CLI 时写关键文档结论>
- <不涉及时写“不涉及第三方文档查询”>

## 4. 已确认的关键选择

- <用户已经确认的方案分叉>
- <如果没有关键分叉，说明为什么没有>

## 5. 总阶段地图

### Phase 1: <阶段名>

- 状态：待细化 | 细化中 | 可编码 | 已完成 | 暂停
- 目标：<阶段目标>
- 范围：<阶段边界>
- 依赖：<依赖其他阶段或外部条件>
- 当前细化程度：粗略阶段 | 文件级 | 函数级 | 可编码

### Phase 2: <阶段名>

- 状态：待细化
- 目标：<阶段目标>
- 范围：<阶段边界>
- 依赖：<依赖其他阶段或外部条件>
- 当前细化程度：粗略阶段

## 6. 编码准备度 review

### 当前细化范围

- Phase: <阶段名或多个阶段>
- 可连续执行范围：<如果只有单阶段，写该阶段；如果多个阶段都成熟，列出阶段范围>

### 当前结论

- 状态：可以进入连续编码 | 需要继续细化 plan | 需要重新澄清目标/阶段边界

### 已足够明确

- <目标 / 文件 / 函数 / 数据流 / 副作用 / 验证中已经明确的内容>

### 仍需细化

- <还不够支撑本阶段一次性编码的内容>

### 已通过问题工具确认的判断

- <用户已确认无偏差的 review 判断>

### 进入本阶段编码前必须确认

- <剩余关键问题；如果没有，写“无”>

## 7. 下一轮细化入口

- 如果继续细化：建议细化 <Phase X>
- 为什么：<该阶段距离可编码还差什么>
- 本轮不细化的阶段：<保持粗粒度的阶段列表>

## 8. 风险与注意点

- <技术风险>
- <边界风险>
- <用户体验风险>
- <验证风险>

## 9. 验证建议

- <自动化测试建议>
- <手动验证建议>
- <回归检查建议>

## 10. 等待用户确认

请确认：

1. 总阶段地图是否仍正确。
2. 当前细化阶段是否选对。
3. 当前阶段是否已经达到本轮编码准备度。
4. 是否允许执行流程按已标记可编码的范围连续实现。
5. 是否仍需对某个阶段、文件、函数、数据流继续细化。
```

模板原则：

- 轻量，便于审批。
- 写调研结论，不写长篇推理。
- 计划目标覆盖完整功能和完整需求。
- 保留总阶段地图。
- 默认用阶段拆解承接大功能，不把大功能缩水成所谓最小边界。
- 把当前批准执行范围细化到可编码；未轮到且不在批准范围内的阶段保持粗粒度。
- 写验证建议，不写测试实现细节。
- 不复制大段源码或官方文档。

## 与现有 Skills 的关系

### clarify-problem

借鉴它的澄清方式：先确认 Claude 和用户说的是同一个目标或现象。

区别是：`clarify-problem` 只澄清问题；`plan-mode-plus` 澄清后会继续做调研、方案分叉确认、阶段级成熟度 review 和 plan 文档生成。

### modular-plan-docs

`modular-plan-docs` 负责已批准计划的项目本地持久化。

如果某个阶段已成熟并批准，可由它落成本地快照，并标明当前可恢复阶段。

### atomic-plan-executor

只借鉴这些规则：

- 复杂任务先调研再计划。
- 关键分叉先问。
- 提问要通俗解释好处和代价。
- 计划轻量，不写完整代码。
- 计划批准不等于编码批准。

不要执行 `atomic-plan-executor` 的这些阶段：

- Approved Plan Snapshot
- 项目本地计划快照
- Implement

### planning-with-files

只借鉴“调研发现要结构化”的思想。

不要创建或维护：

- `task_plan.md`
- `findings.md`
- `progress.md`
- 项目根三文件进度系统

### skill-mining-review

借鉴它的 Skill 写法：触发场景、判断步骤、操作步骤、验证方式、不适用边界要清楚。

不要执行它的日志挖掘职责：

- 不读取 edit logs。
- 不生成 skill candidates。
- 不判断是否升级正式 Skill。

## 反模式

禁止这些行为：

- 用户刚说一个需求，就直接写 plan，不调研、不澄清。
- 把猜测写成“已调研事实”。
- 把未轮到的后续阶段提前写成函数级施工图。
- 把 plan 写成完整代码。
- 每轮 review 只给静态结论，不用问题工具展示判断。
- 用问题工具让用户替 Claude 设计方案，而不是校正 Claude 的 review 判断。
- 计划刚生成就开始编码。
- 创建项目本地 `plans/` 或三文件规划系统。
- 修改 `CLAUDE.md`。
- 把模块化 plan 管理混入本 Skill。
- 把 `atomic-plan-executor` 的执行、checkpoint、doc reconcile 职责混入本 Skill。

## 验证方式

### 结构验证

检查 `SKILL.md` 是否满足：

- frontmatter 有 `name: plan-mode-plus`。
- description 明确触发场景、阶段级细化和负面边界。
- 正文有“只写原生 plan mode 当前指定的计划文件”硬约束。
- 正文有“读取上一版 plan → 按阶段 review → 用问题工具展示判断 → 细化当前阶段”的流程。
- 正文有总阶段地图规则。
- 正文有当前阶段编码准备度 review 规则。
- 正文有每轮 review 多用问题工具的规则。
- 正文没有项目本地计划快照规则。
- 正文没有模块 `CLAUDE.md` 管理规则。
- 正文没有实现、checkpoint、doc reconcile 阶段。

### 行为 dry run

用这些模拟请求检查边界：

1. “先不要写代码，帮我规划一下怎么给登录页加记住登录状态。”
   - 应先澄清和调研。
   - 最终只生成原生 plan mode 当前指定的计划文件。

2. “这个 plan 能不能直接开始写第一阶段？”
   - 应先 review Phase 1 的计划成熟度。
   - 应用问题工具展示判断。
   - 不足则继续细化 Phase 1。
   - 不应把所有阶段都细化到函数级。

3. “继续细化第二阶段，直到能一次性完成编码。”
   - 应读取上一版 plan。
   - 应确认当前细化阶段是 Phase 2。
   - 应用问题工具拆开展示 Phase 2 的目标、文件、函数、数据流、验证缺口。
   - 应继续调研或提问。

4. “第三阶段以后先别细化。”
   - 应保留后续阶段粗粒度地图。
   - 不应提前写函数级施工图。

5. “帮我在项目里建 plans 目录，把计划拆成模块文档。”
   - 不应由 `plan-mode-plus` 承接。
   - 应说明这是模块化 plan 管理 Skill 的范围。

6. “直接把这个 bug 修了，不用计划。”
   - 不应强行触发 `plan-mode-plus`。

7. “规划一下把请求库换成 axios 最新版本。”
   - 应调研现有请求封装。
   - 应查询 axios 当前官方文档。
   - 不应安装依赖或修改 package 文件。

### 边界回归

确认不会发生：

- 创建 `task_plan.md`、`findings.md`、`progress.md`。
- 创建项目本地 `plans/*.md`。
- 修改项目 `CLAUDE.md`。
- 调用写工具修改源码。
- 把 plan 写成完整代码。
- 把所有未来阶段都提前细化到函数级。
- 计划完成后自动进入编码。
