# Plan Implementation Runner

## 目标

`plan-implementation-runner` 负责按已批准且已成熟的计划范围连续执行。

它解决的问题是：旧 `atomic-plan-executor` 把计划生成、本地快照、编码前确认、执行、checkpoint、文档对账揉在一起，且“atomic task / 原子任务”会把阶段实现切得太碎。这个 Skill 只保留执行职责：读取成熟计划，按已批准范围连续推进，必要时逐个写文件节点确认，完成后验证和 checkpoint。

## 一句话边界

只做“已批准成熟计划范围的连续实现编排”，不生成 plan，不补全粗 plan，不创建项目本地计划快照，不做文档对账，不替代实现审查或测试设计。

## 前置条件

使用本 Skill 前必须满足：

- 当前要执行的范围已经由 `plan-mode-plus` 细化到可编码。
- 用户已经批准执行该范围；如果整份多阶段计划都成熟并获批准，应按批准范围连续执行。
- 当前批准范围的不做项清楚。
- 计划中已有足够的文件范围、关键函数、数据流、副作用、错误流和验证方式。

如果发现计划还停留在粗阶段图，或下一阶段没有达到可编码精细度，必须回到 `plan-mode-plus`，不要在 runner 里现场补计划。

## 什么时候使用

当用户表达以下意图时使用：

- “这个阶段计划已批准，开始实现”。
- “继续执行计划里的下一阶段”。
- “从本地计划快照恢复并继续实现”。
- “按这个成熟 plan 开始写代码”。
- “不要拆太碎，按阶段推进”。

## 什么时候不要使用

不要在这些场景使用：

- 用户还在规划或细化阶段，应使用 `plan-mode-plus`。
- 用户要把已批准计划落成本地文档，应使用 `modular-plan-docs`。
- 用户只是要写文件前施工图确认，应使用 `edit-node-blueprint`。
- 用户只是问当前进度，应使用 `execution-checkpoint`。
- 用户要审查已实现代码链路，应使用 `implementation-review`。
- 用户要设计或执行前端测试，应使用 `frontend-test`。
- 用户要求实现后文档对账；当前应单独规划 doc reconcile 类 Skill。

## 工作总流程

固定流程：

```text
确认已批准且当前执行范围可编码
→ 读取计划 / 快照
→ 确认批准范围和不做项
→ 识别批准范围内的写文件节点队列
→ 对每个独立写文件节点执行：edit-node-blueprint → 用户确认 → 写入
→ 运行必要验证
→ 必要时 execution-checkpoint 汇报状态
→ 无阻塞、无偏离且后续阶段成熟时继续执行；否则暂停等待用户决策
```

## 第 1 步：确认当前阶段可编码

执行前先确认：

- 当前批准执行范围是什么：整份计划、某几个阶段，还是单个阶段。
- 用户是否明确批准进入实现。
- 计划是否标明批准范围已达到可编码精细度。
- 不做项是否清楚。
- 验证方式是否清楚。

如果不满足，停止并建议回到 `plan-mode-plus`。

不要为了推进执行而现场补计划。

## 第 2 步：识别写文件节点队列

根据成熟计划，识别当前批准范围内可能的写文件节点队列。

写文件节点是：

- 一次独立 `Edit`。
- 一次独立 `Write`。
- 一组必须在同一次写入中完成、职责紧密相关的函数修改。

识别队列时可以先用普通文本告诉用户当前批准范围预计有几个节点，但不要一次性审批全部节点。必须说明这些节点共同服务于完整功能目标；逐节点确认是为了控制写入风险，不是把执行范围降级成最小版本。

示例：

```text
当前批准范围预计有 3 个写文件节点，它们共同完成已批准的完整功能目标：
1. 修改 A 文件的退出入口。
2. 修改 B 文件的资源清理逻辑。
3. 修改 C 文件的测试断言。
接下来会逐个节点确认施工图；每个节点完成后，在无阻塞和无偏离时继续下一个节点。
```

## 第 3 步：逐个写文件节点确认

每个独立写文件节点在写入前必须遵守 `edit-node-blueprint` 的规则。

核心要求：

- 一个写文件节点一次问题工具。
- 多个独立 Edit 多次问题工具。
- 同一次问题工具里的 tabs/问题只拆解当前节点。
- tabs 要按函数分工动态生成，不套固定模板。
- 用户确认无偏差后，才执行该节点写入。

如果用户指出偏差，先修正施工图，再决定是否继续。

## 第 4 步：执行代码修改

用户确认当前写文件节点后，才可以调用写文件工具。

执行时遵守：

- 只改当前节点确认过的范围。
- 不顺手扩大到未确认文件。
- 不把当前节点边界解释成完整功能缩水。
- 不引入计划外依赖。
- 不绕过测试、lint 或项目已有约束。
- 不跳过安全边界。

如果实现过程中发现计划外约束：

- 停止扩大改动。
- 用 `execution-checkpoint` 说明偏离或阻塞。
- 必要时回到 `plan-mode-plus` 细化。

## 第 5 步：运行必要验证

根据阶段计划运行必要验证。

验证可能包括：

- 单元测试。
- 类型检查。
- lint。
- 构建。
- 前端页面手动验证。
- 现有项目指定命令。

如果是前端 UI 改动，应尽量启动项目并使用浏览器验证真实用户路径。不能验证时要明确说明。

如果需要设计新的前端测试方案，使用 `frontend-test`。

## 第 6 步：连续执行与 checkpoint

默认规则：已批准成熟范围内连续执行，不因为单个阶段完成就硬停。

出现以下情况时使用 `execution-checkpoint` 汇报状态：

- 当前阶段完成，但后续批准阶段仍可继续。
- 执行中断前。
- 出现计划外约束。
- 验证方式降级。
- 依赖、构建方式、模块边界发生变化。
- 用户问“现在进度到哪了”。

checkpoint 默认只是状态汇报，不是阶段门禁。只有阻塞、偏离计划、验证失败、下一阶段未成熟、涉及危险操作，或用户明确要求阶段间确认时，才停止等待用户下一步。

## 与其他 Skills 的关系

### plan-mode-plus

负责计划生成、上一版 plan review、阶段级细化和当前阶段编码准备度判断。

如果 runner 发现计划还不够细，应回到 `plan-mode-plus`，不要自己补计划。

### modular-plan-docs

负责已批准计划的项目本地持久化、索引和归档。

runner 不负责创建项目本地计划文档。

### edit-node-blueprint

负责每个写文件节点前的施工图确认。

runner 在每个独立写文件节点前调用或遵守它的规则。

### execution-checkpoint

负责阶段完成、阻塞、偏离计划或验证降级时的状态记录。

runner 不把 checkpoint 写成长期文档。

### implementation-review

负责实现完成后的链路审查。

runner 不替代 review。

### frontend-test

负责前端测试设计、补充和执行。

runner 可以运行已有验证，但不替代前端测试 Skill 的测试设计职责。

## 连续执行语义

`plan-mode-plus` 反复细化计划的目的，是在编码前消化不确定性。只要计划已经覆盖批准范围、达到可编码精细度，并且用户批准执行，runner 就应尽量连续完成该范围。

阶段边界用于组织实现和汇报进度，不自动等同于用户确认门。阶段完成后，如果下一阶段也在批准范围内且成熟，继续执行；如果下一阶段不成熟或实际情况偏离计划，才暂停。

### skill-mining-review

负责从日志沉淀 Skill。

runner 不做 Skill 挖掘。

## 反模式

禁止：

- 计划还很粗就直接实现。
- 在 runner 中现场补全粗 plan。
- 强制把阶段拆成很多 atomic tasks。
- 已批准成熟计划中每完成一个阶段就默认硬停。
- 把多个独立 Edit 合并成一次提问。
- 写文件前没有展示当前节点施工图。
- 超出当前节点确认范围顺手改其他文件。
- 阶段偏离计划后继续默默执行。
- 用 runner 替代 `implementation-review` 或 `frontend-test`。
- 做 doc reconcile 或长期文档对账。

## 验证方式

### 结构验证

检查 `SKILL.md` 是否满足：

- frontmatter 有 `name: plan-implementation-runner`。
- description 明确“成熟阶段计划已批准后使用”。
- 正文明确不生成 plan、不补全粗 plan、不创建项目本地快照。
- 正文明确每个独立写文件节点前使用 `edit-node-blueprint`。
- 正文明确阶段完成、阻塞、偏离计划或验证降级时使用 `execution-checkpoint`。
- 正文明确不替代 `implementation-review` 和 `frontend-test`。
- 正文避免把 atomic task 作为默认推进方式。

### 行为 dry run

1. “这个阶段计划已批准，开始实现。”
   - 应先确认当前阶段已可编码。
   - 应识别写文件节点队列。
   - 不应重新生成计划。

2. “这个 plan 还只有阶段目标，开始写吧。”
   - 应拒绝直接执行。
   - 应建议回到 `plan-mode-plus` 细化。

3. “这阶段要改三个文件。”
   - 应逐个写文件节点确认。
   - 不应一次问题工具打包三个独立 Edit。

4. “实现完了，检查有没有漏链路。”
   - 应转到 `implementation-review`。

5. “前端页面怎么测？”
   - 应转到 `frontend-test`。
